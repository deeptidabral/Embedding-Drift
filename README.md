# Embedding Drift Monitoring for Fraud Detection

This project demonstrates how to detect and mitigate embedding drift in a dual-layer fraud detection system. It uses the **Sparkov Credit Card Fraud Detection** dataset (1.8M simulated transactions, ~1,000 customers, ~800 merchants) from Kaggle as the primary data source.

The system has two layers. An XGBoost model scores every transaction in sub-10ms for real-time authorization. A RAG+LLM pipeline operates asynchronously after authorization to investigate flagged transactions, supporting human analysts in manual review, chargeback defense, and audit documentation. Embedding drift in the RAG layer and feature drift in the ML model can both silently degrade performance when left unmonitored.

All tools run locally with no external API keys. Embeddings are generated by sentence-transformers (all-MiniLM-L6-v2, 384 dimensions). LLM inference uses Ollama (phi3:mini). Drift detection uses Maximum Mean Discrepancy (MMD) as the sole metric, because dense embedding dimensions are highly entangled and univariate tests (KS, PSI) fail in this setting. Monitoring uses LangSmith for real-time tracing and Evidently AI for periodic HTML drift reports.

## 1: Problem Statement

- Payment processors operate under sub-50ms latency SLAs for authorization. The ML model (XGBoost/LightGBM) alone drives real-time authorization decisions in sub-10ms.
- Transactions in the gray zone (ML score 0.3-0.7), high-value transactions (above $10K), and transactions requiring explainability are flagged for manual review by human analysts.
- A RAG+LLM layer operates **asynchronously after authorization** to support post-transaction investigation. It retrieves historical fraud patterns from a vector database and synthesizes contextual risk assessments. The LLM never blocks a transaction decision.
- Embedding drift in the RAG layer causes retrieval quality to degrade, feeding the LLM irrelevant context for investigation reports. This leads to missed fraud in manual review queues and weaker chargeback defense.
- Feature drift in the ML model (covariate shift, concept drift, target shift) erodes primary scoring accuracy, directly impacting real-time authorization quality.
- Under PCI DSS and SOX, undetected drift in either layer is a control failure.

**Important note on embedding approach.** This project uses text embeddings (sentence-transformers) as a simplified demonstration of drift monitoring concepts. Production fraud detection systems use Deep Entity Embeddings, Autoencoders, or Graph Neural Networks (GNNs) to embed structured tabular data. The drift monitoring framework demonstrated here applies equally to any embedding type.

## 2: Architecture

```
                REAL-TIME AUTHORIZATION PATH (sub-50ms SLA)
                ============================================

Transaction -> ML Model (XGBoost, sub-10ms) -> fraud_score -> Decision Router
                                                                   |
                      +--------------------------------------------+------------------+
                      |                                                               |
               Clear decision                                           Gray zone / High value
               (score < 0.3 or > 0.7)                                                |
                      |                                                               v
                      v                                                Authorization + flag
               Approve / Decline                                       for manual review
                                                                                      |
- - - - - - - - - - - - - - - - - - - - - ASYNC BOUNDARY - - - - - - - - - - - - - - +
                                                                                      |
                ASYNC INVESTIGATION PATH (no latency SLA)                             v
                ==========================================
                                                                       Async queue -> RAG+LLM
                                                                       (embed, retrieve, assess)
                                                                                      |
                                                                                      v
                                                                       Investigation report
                                                                       -> Analyst review queue
```

- **Drift monitoring.** MMD (Maximum Mean Discrepancy) is the sole embedding drift metric. Dense embedding dimensions are highly entangled. Univariate tests (KS per dimension) suffer from massive multiple-testing problems. Cosine distance only captures mean shift. PCA explained variance misses mean shift entirely. Only MMD correctly assesses the full high-dimensional distribution.
- **ML feature drift.** Monitored separately via per-feature PSI, prediction distribution shift, and performance metrics.
- **Graduated alert sequence.** Nominal triggers logging. Warning triggers investigation. Critical triggers fallback and escalation.

## 5: Quick Start

### Prerequisites

- Python 3.10 or higher.
- Ollama installed and running (`ollama serve` in a separate terminal, `ollama pull phi3:mini`).
- No external API keys required for embeddings or LLM. LangSmith requires a free API key for tracing.

### Installation

```bash
git clone https://github.com/deeptidabral/Embedding-Drift.git
cd Embedding-Drift
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

### Download the Sparkov Dataset

```bash
pip install kaggle
kaggle datasets download -d kartik2112/fraud-detection -p data/ --unzip
```

Or download manually from https://www.kaggle.com/datasets/kartik2112/fraud-detection and unzip into `data/`.

### Set Up LangSmith (optional, free)

```bash
export LANGCHAIN_API_KEY="your-key-from-smith.langchain.com"
export LANGCHAIN_TRACING_V2="true"
export LANGCHAIN_PROJECT="fraud-detection-embedding-drift"
```

### Run the Full Pipeline

```bash
python examples/full_pipeline_live.py
```

This runs all 9 steps end-to-end: data loading, embedding generation, ML scoring, async LLM investigation, MMD drift detection, Evidently reports, LangSmith tracing, and visualization generation.

### Run Individual Examples

```bash
python examples/fraud_detection_demo.py
python examples/drift_monitoring_demo.py
python examples/evidently_drift_report_demo.py
```

### Run Tests

```bash
pytest tests/ -v
```

## 8: Technology Stack

| Component | Technology | Purpose |
|-----------|------------|---------|
| ML Scoring | XGBoost, scikit-learn | Primary fraud scoring (sub-10ms, real-time authorization). |
| Embeddings | sentence-transformers (all-MiniLM-L6-v2) | Transaction text to 384-dim vectors for RAG layer. Local, no API key. |
| LLM Inference | Ollama (phi3:mini) | Async post-transaction investigation for flagged transactions. Local, no API key. |
| Vector Search | In-memory numpy vector search | Storage and retrieval of transaction embeddings via cosine similarity. |
| Drift Detection | MMD (custom implementation) | Sole embedding drift metric. Pooled RBF kernel, permutation test. |
| Drift Reporting | Evidently AI | Periodic HTML drift reports with PCA-projected embedding analysis. |
| LLM Observability | LangSmith | Async investigation pipeline tracing, custom feedback scores, dashboards. |
| Data Drift | Custom (concept, covariate, target shift) | ML model input feature drift detection. |
| Alerting | PagerDuty, Slack webhooks | Drift alert delivery. |
| Data Processing | Pandas | Transaction data manipulation and feature engineering. |
| Visualization | Matplotlib, Seaborn | Embedding space, drift metrics, pipeline architecture charts. |
